## Analysis Scope & Entry Points
- Overlay chunk interaction via `updateSenseiMeditationOverlay` in `ui.ts`
- Chunk switching hook `window.switchToChunk` in `index.tsx`
- Response orchestration `generateNextSenseiResponse` in `index.tsx`
- Learner model updates in `updateLearnerModel` within `adaptiveEngine.ts`
- Curriculum progression checks in `advanceCurriculumState` inside `curriculum.ts`

## Static Execution Trace
1. Brand icon opens overlay and `updateSenseiMeditationOverlay` renders chunk cards/actions.
2. User activates a chunk control which calls `window.switchToChunk(targetIndex)`.
3. `window.switchToChunk` ensures teaching plan presence, resets state sets, updates UI, and invokes `generateNextSenseiResponse('', true)`.
4. `generateNextSenseiResponse` fetches current item, obtains analysis when needed, and forwards data to `updateLearnerModel`.
5. `updateLearnerModel` normalizes analysis, updates KC mastery, `contentPointsCoverage`, and chunk coverage sets inside `curriculumState`.
6. `generateNextSenseiResponse` evaluates advancement by calling `advanceCurriculumState` and refreshes UI/notepad state.
7. `advanceCurriculumState` inspects `coveredPointsInCurrentChunk` and `pointsToRevisitInCurrentChunk` to move to the next chunk or phase and resets tracking structures.
8. Updated state propagates back to `updateSenseiMeditationOverlay` for refreshed overlay presentation.

## Architectural Insights
- Chunk mastery is inferred from `curriculumState.coveredPointsInCurrentChunk` combined with `learnerModel.contentPointsCoverage` high-water marks, making manual overrides require synchronized updates to both structures.
- KC mastery for a phase accumulates incrementally through `updateLearnerModel` using each teaching point's `kcValue`, meaning forced completion must award residual KC deltas while respecting the 1.0 cap.
- Overlay rendering is stateless beyond `curriculumState`, so persistent understanding flags will need dedicated storage within curriculum/learner state for cross-session fidelity.
