## Static Execution Trace (success path)
1. `displayMessage` (sensei message) parses markdown and locates `<pre><code class="language-mermaid">` nodes.
2. For each block, `displayMessage` requests `mermaidManager.render(id, code)` to obtain SVG markup.
3. On success, `renderMermaidThumbnailWithTheme(preElement, svg, currentTheme, code)` replaces the `<pre>` with themed thumbnail and attaches lightbox behavior.
4. The surrounding `messageText` retains following paragraphs (including italic annotation) with default bubble width.

Alternate fallback trace when rendering fails:
1. `displayMessage` applies `applyUniversalQuoteFix`/`applyBacktickFix` then retries `mermaidManager.render`.
2. If retry fails, it calls `attemptMermaidFix` (LLM-assisted) and, upon success, re-enters the success path at step 3; otherwise displays error block.

## Key Architectural Insights
- Annotation text currently remains as generic paragraph nodes under `.message-text`, inheriting full bubble width (`flex:1`) and `text-align: justify`, so it is not visually scoped to the diagram footprint.
- `.mermaid-thumbnail--horizontal` hard-codes 75% width while vertical diagrams use `width: fit-content`; there is no paired container to align accompanying text, suggesting we need a wrapper or adjacent-style rule per orientation.
- Prompt directive only requires a “brief” annotation, leading to shallow descriptions; no other prompt text augments this, so extending `MERMAID_GENERATION_GUIDELINES` is the lever for richer explanations.
- Thumbnail renderer has no knowledge of annotations, but by moving or cloning adjacent annotation nodes we can couple them without disturbing existing lightbox logic.
- Error-handling path (quote fixes + AI recovery) must remain untouched; our changes must slot into the success branch after SVG replacement to avoid destabilizing recovery logic.
