# Mission State Checkpoint — Header Chunk Overlay Access

## Entry Point & Scope
- Entry point: User click on `.weighted-segment.status` (middle header segment labeled Current Focus).
- Scope files: `index.html` (header layout), `ui.ts` (initializeUI, meditation overlay logic, hover state), `index.tsx` (`window.switchToChunk` integration used by overlay chunk cards).
- Reviewed the relevant sections of each file to confirm event hooks, overlay rendering, and curriculum state access patterns.

## Static Execution Trace
1. User clicks the middle header segment `.weighted-segment.status`.
2. Planned handler `showMeditationOverlayFromStatus` retrieves `window.curriculumState`, sets `meditationHoverState.showAllChunks = true`, and calls `updateSenseiMeditationOverlay(curriculumState, true)`.
3. `updateSenseiMeditationOverlay` rebuilds the overlay, populates chunk cards, and exposes chunk toggles while ensuring the overlay becomes visible.
4. Clicking a chunk card triggers the inline handler which calls `window.switchToChunk(chunkIndex)`.
5. `window.switchToChunk` resets chunk tracking structures, refreshes UI via `updateCurriculumDisplay` and `updateSenseiMeditationOverlay`, updates KC progress, and invokes `generateNextSenseiResponse`.

## Key Architectural Insights
- Overlay visibility is orchestrated through `meditationHoverState`; triggering from the header must manipulate this state to present the all-chunks view without relying on brand hover.
- `updateSenseiMeditationOverlay` assumes `meditationOverlay` DOM nodes exist and rebinds handlers on every call; invoking it directly on status clicks will reuse existing rendering logic.
- `window.switchToChunk` performs substantial state mutation and LLM calls; the status-triggered overlay should only expose selection, not auto-switch chunks.
- Overlay hide timers depend on hover flags; status-triggered launches must keep the overlay open long enough for users to interact with chunk cards.
