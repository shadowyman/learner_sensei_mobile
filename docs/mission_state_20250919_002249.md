# Mission State Checkpoint (2025-09-19 00:22:49)

## Analysis Scope & Entry Points
- Entry Trigger: User hover/click interaction on `.weighted-segment.brand` (brand icon) which exposes the Sensei meditation overlay.
- Primary Modules in Scope: `ui.ts` (overlay rendering & interactions), `index.tsx` (UI initialization and global curriculum state exposure), `curriculum.ts` (chunk & phase state management), `adaptiveEngine.ts` (learner model content point tracking).

## Static Execution Trace
1. `initializeUI()` (index.tsx → ui.ts) during app bootstrap.
2. `setupBrandHoverMeditationOverlay()` binds brand segment hover listeners.
3. Runtime event: brand interaction invokes `updateSenseiMeditationOverlay(curriculumState, true)`.
4. `updateSenseiMeditationOverlay()` assembles DOM for current/all chunk view and calls `showMeditationOverlay()`.
5. `showMeditationOverlay()` enables pointer events, hooks hover guards, and animates overlay (Anime.js fallback to CSS).
6. Overlay exit path uses `hideMeditationOverlay()` when hover guard releases control.

## Key Architectural Insights
- Overlay relies on `window.curriculumState` being current; any chunk switching must re-run `updateCurriculumDisplay` (or equivalent) so overlay shows updated state.
- Chunk progress tracking spans both `curriculumState` sets and `learnerModel` structures (e.g., `contentPointsCoverage`, `awardedKcForPhasePoints`), so “restart chunk” must clear/realign both.
- Navigation helpers already exist for concept jumps and phase resets; chunk switching should mirror their reset patterns without regenerating entire teaching plans unless necessary.
- UI module currently has no bridge back into curriculum control; new chunk-click handling will require an exposed index.tsx API (likely `window` binding) to avoid circular imports.
