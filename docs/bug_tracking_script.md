# Bug Tracking Script

| Bug ID | Title | Status | Reported On | Reporter | Description | Notes |

| 1 | Path alias imports fail to resolve | Open | 2025-09-26 | Codex | `resolveImport` (`scripts/analyze.ts:255-270`) only accepts relative specs, so any module resolved via `tsconfig.json` `paths` (e.g., `'@/foo'`) or node module resolution is dropped; downstream import graph and call graph omit those files entirely. | Replace custom path math with TypeScript’s resolver (`ts.resolveModuleName` or `program.getResolvedModuleWithFailedLookupLocationsFromCache`), normalize the resolved filename via `rel`, and guard with `isProjectSource` so alias-based imports contribute edges. |
| 2 | DOM selector extraction misses common patterns | Open | 2025-09-26 | Codex | `selectorsFromHtml` relies on double-quote regexes, so HTML fragments with single-quoted attributes, unquoted values, or nested markup yield no selectors, undermining `--dom-index`. | Swap the regex logic for an HTML parser (e.g., `parse5.parseFragment`), walk element nodes, and emit selectors for every `id` / `class` attribute regardless of quoting; update dependencies accordingly. |
| 3 | Stable ID map cleared during focused recollect | Open | 2025-09-26 | Codex | Clearing `functionStableIdById` before running a focused `collectFunctions` pass removes stable IDs for functions outside the include set, so edges targeting them lose their `toStable` values, corrupting the call graph. | Preserve the cleared stable-ID map across the focused recollect by saving it (e.g., `const savedStableIds = new Map(functionStableIdById)`) and merging it back afterward for entries missing in the rebuilt map. |
| 4 | stripNonNull hides receivers behind casts | Open | 2025-09-26 | Codex | Many call-site analyses invoke `stripNonNull`, which only removes `!`. When receivers are wrapped in parentheses, `as`/`<type>` assertions, or other wrappers, the analyzer never reaches the underlying identifier, so member lookups and `this` propagation fail. | Deprecate `stripNonNull` in favor of the more thorough `unwrap`; replace its call sites (e.g., property and element access handling) with `unwrap`, and optionally rename `unwrap` to `unwrapExpression` for clarity. |
| 5 | Global exposure edges lack source node | Open | 2025-09-26 | Codex | `scanGlobalExposures` adds edges from `'global::exposed'` without creating a matching function entry, so downstream consumers see edges whose source node is missing. | Insert a synthetic function record (`id`/`stableId` `'global::exposed'`) into `funcs` (and seed `functionStableIdById`) before emitting artifacts, or equivalently filter these edges out of exports if consumers cannot handle synthetic nodes. |
| 6 | Dead recordEdge initializer branch | Open | 2025-09-26 | Codex | `recordEdgeForInitializer` guards on `ts.isMethodDeclaration(init)` even though `init` is always a `ts.Expression`; the block never runs and signals a misunderstanding of callback handling. | Delete the unreachable branch and rely on the existing `visitObjectLiteralArg` handling for inline method callbacks to keep the code path clear. |
| 7 | TC-resolved calls skip callback/DOM analysis | Open | 2025-09-26 | Codex | In `analyzeFile` the type-checker branch exits via `return` after linking the callee, so argument scanning, DOM/event instrumentation, and side-effect tagging never run for those call sites. | Restructure the call-expression logic to track whether the callee was resolved (e.g., `linkedCallee`) but continue through the rest of the analysis instead of returning early. |
| 8 | --no-typechecker still uses checker | Open | 2025-09-26 | Codex | Even with `--no-typechecker`, `program.getTypeChecker()` is invoked and many helpers (e.g., `recordEdgeForInitializer`, `scanGlobalExposures`) call `checker.*`, negating the flag and hurting performance. | Only instantiate the checker when `useTypeChecker` is true and guard every checker-dependent helper with fallback heuristics when the flag is off. |
| 9 | Filtered runs add excluded files as entry candidates | Open | 2025-09-26 | Codex | `collectImportGraph` inserts excluded files with empty dependency lists; `selectEntryCandidates` then treats every zero–fan-in file as an entry during `--include` runs. | Skip excluded files entirely (or filter them when computing fan-in/out) so entry candidate detection reflects only in-scope modules. |
| 10 | Static namespace imports lack lookup | Open | 2025-09-26 | Codex | `collectAliases` omits `import * as ns` bindings, leaving static namespace receivers unresolved when the type checker is unavailable. | Extend alias collection to capture namespace imports and consult them alongside dynamic `nsImports` when linking property accesses. |
