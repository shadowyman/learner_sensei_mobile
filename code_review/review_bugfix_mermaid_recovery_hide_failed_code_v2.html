PR Review Context

  1. User-visible bug fix: after five failed Mermaid recovery attempts, Sensei displayed the full diagram codeblock inside the chat bubble. This exposed potentially noisy or sensitive content and cluttered the UI. The root cause was that failure branches in ui.ts and SelectionSensei appended a <pre><code> block on recovery exhaustion. This change removes the raw code from the DOM entirely and renders a standardized red-box message only, preserving the existing error styling for clear user feedback. Internally, I added debug logs tagged [MERMAID_FAILOVER] that capture the complete diagram string alongside path context (messageId or selection). These logs complement the existing [MERMAID_RECOVERY] entries for end-to-end traceability. The scope touches two callsites in ui.ts (initial render and phase-2 reprocess) and the SelectionSensei handler, ensuring parity across surfaces. No changes were made to the rendering pipeline, theme manager, or recovery algorithm, so performance and behavior remain stable except for the post-exhaustion UI. Risk is low: the refactor reduces code exposure, and the fallback container still uses the same CSS class for consistent styling. Manual validation will trigger a failing diagram to verify that only the bracket message appears and that debug logs record the full code. This fix also aligns with security and UX goals by avoiding inadvertent content leakage in the transcript.
  2. Follow-up change: persist recovery outcomes in saves by normalizing the in-memory raw Sensei text. When recovery succeeds, the code replaces the original mermaid fence with the corrected diagram inside the raw message map; when recovery exhausts, it replaces the fence with the single bracketed error line. This ensures save files reflect the on-screen outcome, and restores do not need to retry fixes unnecessarily. Implementation adds a small helper to surgically replace the matching mermaid fence (exact-match first, then a generic fallback) and wires it in both the initial render path and the phase-2 reprocess path right where recovery results are known. SelectionSensei is intentionally unchanged for persistence since it is not part of the chat transcript. Risk assessment: Low. The change only updates the raw text map; display logic and rendering remain unchanged. If the exact-match replacement fails due to whitespace variance, a generic replacement targets the first mermaid fence in the message, which mirrors block order. Validation: Will trigger both a successful recovery and a forced failure to confirm the raw map updates, review logs for [MERMAID_FAILOVER] messages, and verify that subsequent saves contain the corrected diagram or the single error line respectively.
